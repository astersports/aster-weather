# @aster/weather — End-to-End Efficiency & Enhancement Report
**Date:** 2026-07-16  ·  **Scope:** post-convergence audit across `aster-weather` (package), `aster-sports`, `st-patricks-armonk`, `aster-studio`  ·  **Operator questions answered:** (1) Is the end-to-end setup as efficient as possible? (2) What to add for weather MONITORING and NOTIFICATION.

> All findings below are verified in-repo (file:line grounded, adversarially checked). Severities reflect reviewer recalibration where the original label ran hot; the recalibrated call is stated inline. Five rejected findings (mostly the "v0.2.0 blocks the API surface" class) are excluded — the public API surface is byte-identical from v0.2.0 through v0.4.0, so `getNowcast`/`getCurrentWeather`/`getDailyForecast`/`observedAt` are all callable at the pinned v0.2.0. That fact reshapes several recommendations below.

---

## 1. Efficiency verdict

**Direct answer: No — not yet as efficient as it could be, but the gaps are bounded and mostly one logical class: the same coordinate gets fetched more than once because the fan-out and the parallel clients each own their own request + cache.** The convergence did real work (one shared engine, cache-batched per-schedule fetches, tree-shaken icon chunk), but three live inefficiencies remain and one is a regression.

### What is already efficient (do not "fix")
- **Per-schedule cache batching works as intended.** `st-patricks-armonk/server/weather/forecast.ts:154-184` (`getWeatherForEvents`) pre-warms with ONE `fetchForecast()` then `Promise.all`s the per-event calls, all hitting the warm 60-min cache; in-flight dedup (`aster-weather/src/cache.ts:39-41`) collapses the concurrent first-calls. **1 network fetch for N same-coord events.** — *P3 confirm-good.*
- **The heavy SVG icon chunk is pulled only where SVGs render.** `@aster/weather` ships `sideEffects:false` + a separate `./icons` subpath; the ~515-LOC colorful/motion icon set tree-shakes out of the emoji consumers. Only `st-patrick` client imports `@aster/weather/icons` (`WeatherIcons.tsx:20`). aster-sports (emoji) and studio (no UI) pull neither. — *P3 confirm-good.*
- **The 3-decimal `coordKey` (`helpers.ts:49` `toFixed(3)`) means same-venue events share one cache entry.** Keep it.

## 3. Monitoring / observability roadmap

**Direct answer: the weather subsystem is the one user-facing area deliberately excluded from the app's own observability discipline. There is currently no programmatic way for an operator to know Open-Meteo is failing for a venue — the engine converts every failure to `[]`/`null`/stale before any consumer regains control.** The most important single unlock is a package-level `onError` hook; everything else is downstream of it.

## 4. Weather-notification enhancements (the headline)

**Direct answer to "what to add for notification": two features, both grounded in engine fields that already exist and both reusing the aster-sports Stream A/B auto-notification plumbing verbatim.** Today Frank only learns of bad weather by manually eyeballing `RainAlertBanner` — nothing routes a rain-out-risk game to admins, and the freshest signal the package produces (`getNowcast`) reaches no surface at all.

### N1 — Severe-weather game advisory: auto-DRAFT into the Radar *(lead / headline ask)*
